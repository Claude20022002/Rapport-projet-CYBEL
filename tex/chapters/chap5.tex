\section{Méthodologie et planification}
\label{chap:methodologie_planification}

Dans le cadre de notre stage, nous avons adopté une démarche
méthodique combinant \textbf{rétro-ingénierie incrémentale non destructive}
et \textbf{développement logiciel itératif mock-first}. Cette approche répond
directement à l'absence de documentation constructeur et à la disponibilité
limitée du robot physique au laboratoire HESTIM.

Ce chapitre détaille notre mode de travail, la planification retenue, les
outils employés, les premières découvertes sur le robot, la documentation
produite, les apports du projet ainsi que les difficultés rencontrées durant
la phase d'investigation.


\subsection{Mode de travail et organisation}

Le stage a réuni deux stagiaires aux \textbf{périmètres distincts} : la
plateforme CYBEL (contenu technique du présent rapport, réalisée par
Claudia LUSAMOTE KIMFUTA sur les branches \texttt{main, face presence et promote kisok poi main}) et un module
\textbf{chatbot} développé en parallèle par Ghassane KHALIFI sur la branche
\texttt{chatbot-cybel-projet-2026}, hors périmètre des chapitres de
réalisation.

Pour la partie CYBEL, le mode de travail a reposé sur une démarche
\textbf{incrémentale et autonome} : exploration documentaire, scripts Python
de validation dans le dépôt \texttt{cybel}, développement mock-first puis
tests terrain lorsque le robot était disponible.

Notre encadrante pédagogique, Mme~\textsc{MESKINI}, notre encadrant en
entreprise, M.~\textsc{TULA}, ainsi que les membres du laboratoire nous ont
accompagnés par des réunions régulières et des comptes rendus journaliers et
hebdomadaires. Pour structurer l'avancement, nous avons
maintenu un planning détaillé et centralisé nos hypothèses, blocages et
solutions dans le dépôt Github du projet.

\subsubsection{Démarche de découverte protocolaire}

Chaque hypothèse sur le protocole du robot a suivi le cycle suivant :
\begin{enumerate}
    \item \textbf{Observation passive} : écoute MQTT et introspection
    \texttt{rosbridge} avant toute publication ;
    \item \textbf{Test isolé} : script Python dédié dans \texttt{scripts/} ;
    \item \textbf{Vérification de l'effet réel} : une commande acceptée par
    \textit{rosbridge} n'est pas forcément exécutée sur le châssis ;
    \item \textbf{Intégration SDK} : portage dans \texttt{MockRobot} puis
    \texttt{RealRobot} ;
    \item \textbf{Documentation} : mise à jour de \texttt{docs/} et des constantes
    dans \texttt{sdk/constants.py}.
\end{enumerate}

\subsection{Planning du projet}

Afin d'assurer une progression cohérente tout au long du projet et de respecter
les objectifs fixés, un planning prévisionnel a été établi dès le début sous la
forme d'un diagramme de Gantt. Celui-ci a permis de répartir les différentes
activités, d'identifier les dépendances entre les tâches et de suivre
l'avancement.

Le déroulement a été organisé en quatre grandes phases successives qui sont:

\begin{enumerate}
    \item La première phase, consacrée à la \textbf{connectivité}, avait pour objectif de
    comprendre l'architecture matérielle et réseau du robot : analyse de la
    topologie, identification des équipements et balayage des ports ouverts.

    \item La deuxième phase concernait l'\textbf{investigation du système}. Des travaux de
    rétro-ingénierie ont permis d'identifier les mécanismes de communication :
    exploration \texttt{rosapi}, analyse MQTT passive, étude du TTS via ADB et
    développement du pont Android \texttt{CybelTTSBridge}.

    \item La troisième phase correspond au \textbf{développement de la plateforme CYBEL} :
    SDK Python (mock et réel), backend FastAPI avec WebSocket, interface opérateur,
    moteur de visite guidée (\texttt{TourEngine}), kiosque visiteur et déploiement
    Termux sur la tablette.

    \item Enfin, la quatrième phase est dédiée à la \textbf{validation du système} : essais
    de navigation terrain, ajustement des coordonnées du parcours
    \texttt{lab\_tour.json}, tests d'intégration et rédaction du présent rapport.
\end{enumerate}

\subsection{Outils, langages et technologies utilisés}

\subsubsection{ROS (Robot Operating System)}

Le châssis du robot exécute ROS pour la localisation SLAM, la planification
(\texttt{move\_base}) et l'exécution des trajectoires. CYBEL n'accède pas à
ROS natif (C++/Python ROS) mais via \textbf{\textit{rosbridge}} pont WebSocket
JSON sur le port 9090.

\subsubsection{FastAPI}

FastAPI constitue le backend principal sur poste opérateur : API REST, WebSocket
\texttt{/ws/telemetry}, routers (\texttt{robot}, \texttt{navigation}, \texttt{tour},
\texttt{speech}, etc.) et configuration via \texttt{pydantic-settings}
(\texttt{ROBOT\_MOCK}, \texttt{ROBOT\_HOST}). Sur tablette Termux, \textbf{Starlette}
(\texttt{cybel\_lite.py}) remplace FastAPI car \texttt{pydantic-core} ne compile
pas sur Python~3.13 Termux.

La topologie réseau s'est révélée dual-stack : le châssis est joignable en

\texttt{eth0} sur \texttt{192.168.20.0/24}. Depuis Termux, \textit{rosbridge}
est accessible sur \texttt{192.168.20.22:9090} et non sur \texttt{10.42.0.1}.

\subsubsection{Zenmap}
Zenmap (figure~\ref{fig:zennmap}) est l’interface graphique officielle de Nmap, conçue pour rendre l’exploration et l’audit des réseaux plus accessibles. Elle permet de lancer des scans sans retenir toutes les options complexes de la ligne de commande, en offrant une présentation claire des résultats sous forme de tableaux et de graphiques. Grâce à ses profils de scan enregistrables, Zenmap facilite la répétition de tâches courantes et constitue un outil pratique pour les administrateurs réseau, les étudiants et les professionnels de la cybersécurité.

JADX est un outil open source qui a permis de désassembler et de décompiler les fichiers APK du constructeur (issus de leurs applications Android) en code Java lisible. Il nous a été particulièrement utile pour le développement, la recherche et l’analyse de la structure interne des applications du constructeur, afin de comprendre leur fonctionnement et de vérifier certains comportements des APK. Grâce à son interface graphique et à ses options en ligne de commande, JADX a facilité l’exploration du bytecode Android et la conversion en code source plus compréhensible.

\subsubsection{Termux}
Termux est une application Android qui fournit un terminal Linux complet directement sur smartphone ou tablette. 
Elle permet d’exécuter des commandes Unix, d’installer des paquets via son propre gestionnaire, et d’accéder à des outils de développement ou de sécurité comme Python, Git ou SSH. 
Grâce à son environnement minimaliste mais puissant, Termux est utilisé aussi bien pour l’apprentissage de Linux que pour l’administration système ou le prototypage rapide sur mobile.

\subsection{Proposition de documentation technique}

Au fil du projet, nous avons structuré une documentation technique dans le
répertoire \texttt{docs/} du dépôt, complétée par des scripts reproductibles
dans \texttt{scripts/}. Les guides principaux sont :

\begin{itemize}
    \item \texttt{ROBOT\_CONNECTION.md} topologie réseau et paramètres de connexion ;
    \item \texttt{INTERFACE.md} interface opérateur ;
    \item \texttt{TTS\_BRIDGE.md} pont de synthèse vocale Android ;
    \item \texttt{VISITOR\_KIOSK.md} et \texttt{TERMUX\_DEPLOY.md} kiosque et déploiement embarqué ;
    \item \texttt{labo/TERRAIN.md} procédure de validation terrain.
\end{itemize}

Les diagrammes Mermaid (architecture, séquences, cas d'utilisation) sont
versionnés dans \texttt{docs/Sujet de stage/diagrammes/} et exportés en PNG
pour le rapport. Cette documentation constitue la référence opérationnelle pour
toute reprise du projet au laboratoire.


\subsection{Apports du projet}

Le projet CYBEL apporte à HESTIM une \textbf{plateforme tierce de commande et
d'interaction}, indépendante de l'application propriétaire du constructeur.
Les apports concrets sont :

\begin{itemize}
    \item un protocole de communication ROS/\textit{rosbridge} documenté et
    intégré dans un SDK Python testable en mode simulé ;
    \item une interface opérateur web (carte SLAM, LiDAR, téléopération, visite
    guidée, accueil vocal) ;
    \item un kiosque visiteur bilingue déployable sur la tablette du robot ;
    \item un canal TTS Android (\texttt{CybelTTSBridge}) absent du protocole ROS ;
    \item une base de connaissances laboratoire (\texttt{knowledgeV2-lab.json})
    transformée en parcours navigable (\texttt{lab\_tour.json}).
\end{itemize}

\subsection{Conclusion}

La méthodologie adoptéeobservation, vérification, simulation puis intégration
s'est avérée indispensable face à un système fermé et partiellement documenté.
Le planning en quatre phases a été globalement respecté, avec une avance sur le
déploiement tablette et un retard acceptable sur la validation navigation
complète du parcours laboratoire. Les outils retenus (Python/FastAPI,
TypeScript/Vite, \textit{rosbridge}, ADB, Termux) forment un écosystème
cohérent pour un stage à contraintes matérielles fortes. Le
chapitre suivant détaille la réalisation concrète de cette architecture.
